## 2026-07-22 — 修复文件夹浏览器无法进入子目录（NewPlaylistDialog）

### 问题

- **现象**：批量导入播放列表的文件夹选择弹窗中，点击盘符（C:/D:/E:）没有任何反应，无法进入下一级目录
- **根因**：`NewPlaylistDialog.vue` 的 `onClickEntry` 函数中，文件夹模式（`fileMode === 'folder'`）下点击文件夹 item 只执行 `selectedPath.value = entry.path` 然后 return，**不调用 `loadBrowserDir` 进入子目录**。导致用户无法浏览到目标子目录

### 修复

- `onClickEntry` 函数：文件夹模式下点击文件夹也进入子目录（移除 folder 模式的 return 分支），与单文件模式行为一致
- 模板：文件夹模式下文件夹 item 也显示进入箭头（`ArrowRight`），提示可点击进入
- 模板：移除文件夹模式下基于 `selectedPath` 的高亮（不再通过点击选中，改由「选择此文件夹」按钮确认）
- 描述文案：文件夹模式提示改为"点击文件夹进入下级，到达目标目录后点「选择此文件夹」确认"

### 交互逻辑（修复后）

| 模式 | 点击文件 | 点击文件夹 |
|---|---|---|
| 单文件模式（single） | 选中 m3u 文件并关闭弹窗 | 进入子目录 |
| 文件夹模式（folder） | — | 进入子目录（通过「选择此文件夹」按钮确认当前目录） |

## 2026-07-22 — 网易云下载功能 + NCM 解密（netease_music 1.1.0 / 前端 1.5.0）

### 需求

- **用户需求**：网易云下载功能，如果下到的是 ncm 加密格式，需要进行解密
- **解密参考**：`音乐解锁DecrypeMusic-main`（从 worker.js 逆向 NCM 格式与算法）

### 设计要点

#### NCM 解密模块（`plugins/netease_music/eta_netease/ncm_decrypt.py` 新建）

- NCM 文件格式：Magic(8B `CTENFDAM`) + Gap(2B) + RC4 Key(4B len + AES-ECB encrypted) + Metadata(4B len + AES-ECB encrypted) + Cover(5B gap + 4B size + image) + Gap(13B) + Audio(XOR encrypted)
- AES-128-ECB 密钥（从 worker.js 逆向）：
  - Core key: `687a4852416d736f356b496e62617857`
  - Metadata key: `2331346c6a6b5f215c5d2630553c2728`
- 非标准 RC4 密钥流生成（无 drop-256），直接生成 256 字节 keystream
- `decrypt_ncm(data) -> NcmResult`：返回 audio_data、fmt、music_name、artists、album、album_pic_url、cover_data
- `decrypt_ncm_file(file_path) -> NcmResult`：从文件读取后解密

#### 下载任务数据库（`plugins/netease_music/eta_netease/database.py` 新建）

- 独立 SQLite `data/netease_music.db`（与主应用数据库分离）

#### 数据模型（`plugins/netease_music/eta_netease/models.py` 新建）

#### 下载器（`plugins/netease_music/eta_netease/downloader.py` 新建）

- 后台线程 + DownloadTask 模型（参考 asmr_one 架构）
- 流程：获取下载URL → 下载文件 → 检测ncm格式 → 解密 → 写入元数据/封面 → 触发扫描 → 添加收集箱
- 支持本地节点和远程节点（通过 `eta_shared.node_client`）
- ncm 解密后自动写入元数据（musicName、artist、album、cover）
- `_is_ncm_file(data)`：检查魔数 `CTENFDAM`
- `start_download_task(task_id)` / `cancel_download_task(task_id)`

## 2026-07-22 — 播放列表文件夹功能（1.5.0）

### 需求

- **用户需求**：播放列表添加文件夹概念。导入播放列表时，如果按文件夹导入，则保留文件夹内的层级关系导入到播放列表树中
- **核心设计**：
  - 文件夹是播放列表之间的树形组织容器（不是列表内部的层级）
  - 导入文件夹 `/Music/Artist1/Album1/playlist.m3u` 时，创建 `Artist1 → Album1 → playlist` 的树形结构
  - 使用独立文件夹实体表（自引用树形模型），非 path 字符串
  - 节点播放列表和客户端播放列表两者都支持文件夹
  - 支持导入 + 手动管理（创建/重命名/删除/拖拽移动）
  - 现有无文件夹归属的播放列表显示在根级
  - 导入时从选择的文件夹下一层开始创建层级
  - 删除非空文件夹时递归删除内容

### 设计要点

#### 数据模型

1. **节点侧 PlaylistFolder**（`node/eta_node/models.py`）
   - 自引用树形：`parent_id → playlist_folders.id`，`ON DELETE CASCADE`
   - `owner_id` 外键到 users，`deleted_at` 软删除，`version_stamp` 增量同步
   - `Playlist.folder_id` 外键到 `playlist_folders.id`，`ON DELETE SET NULL`（文件夹删除时播放列表回到根级）

2. **客户端侧 ClientPlaylistFolder**（`web/backend/eta_web/client_playlists/models.py`）
   - 全局的（无 owner_id），硬删除（无 deleted_at/version_stamp）
   - `ClientPlaylist.folder_id` 外键，`ON DELETE SET NULL`

3. **数据库迁移**（`node/migrate_db_150.py`）
   - 创建 `playlist_folders` 表 + `playlists.folder_id` 列
   - 客户端侧通过 `_auto_migrate()` 自动 ALTER TABLE

#### 后端 API

1. **节点侧文件夹 CRUD**（`node/eta_node/routers/playlist_folders.py`）
   - `GET/POST /api/playlist-folders`，`PUT/DELETE /api/playlist-folders/{id}`
   - 循环检测：`_is_descendant()` 防止将文件夹移动到自己的子文件夹下
   - 递归软删除：`_collect_descendants()` 收集所有后代 id
   - 移动时 `parent_id=0` 表示移到根级

2. **客户端侧文件夹 CRUD**（`web/backend/eta_web/client_playlists/folder_routers.py`）
   - `GET/POST /api/client-playlist-folders`，`PUT/DELETE /api/client-playlist-folders/{id}`
   - 逻辑与节点侧类似，但无权限检查（全局）、硬删除

3. **m3u 导入层级保留**（`node/eta_node/m3u_importer.py`）
   - `_collect_m3u_files` 返回 `(m3u_path, playlist_name, folder_parts)`，folder_parts 是 m3u 相对于导入根目录的父目录路径组件

#### 前端

## 2026-07-21 — 修复 Jenkinsfile 部署阶段 docker pull 失败被静默吞掉的问题

### 需求

- **背景**：1.4.1 Jenkins build #41 显示"部署成功"，但生产环境实际运行的仍是旧版本前端（HTML 引用的 JS hash 未更新）
- **根因**：`execCommand` 使用 `&&` 与 `|| true` 链式连接，shell 的 `&&` 和 `||` 是左结合的，等价于 `((((docker pull && docker stop) || true) && docker rm) || true) && docker run`
- **后果**：`docker pull` 失败时（如 Registry 不可达），`|| true` 让整个子表达式返回成功，继续执行 `docker rm` + `docker run`，由于本地仍有旧的 `latest` tag，容器用旧镜像启动，sshPublisher 报告"部署成功"但实际未更新

### 修复

将 `docker stop` 和 `docker rm` 的容错逻辑包在子 shell `( ... || true )` 中，使其失败不影响外层 `&&` 链对 `docker pull` 和 `docker run` 失败的传递：

```bash
# 修复前
docker pull ${IMAGE_LATEST} && docker stop etamusic 2>/dev/null || true && docker rm etamusic 2>/dev/null || true && docker run ...

# 修复后
docker pull ${IMAGE_LATEST} && (docker stop etamusic 2>/dev/null || true) && (docker rm etamusic 2>/dev/null || true) && docker run ...
```

修复后的失败传递：
- `docker pull` 失败 → 整个命令失败，Jenkins 部署阶段报错
- `docker stop` / `docker rm` 失败（首次部署容器不存在）→ 子 shell 返回 true，继续执行
- `docker run` 失败 → 整个命令失败，Jenkins 部署阶段报错

## 2026-07-21 — 修复已删除远程节点残留显示问题（1.4.1）

### 需求

- **生产环境反馈**：节点管理页与工作台在删除远程节点后仍显示离线节点，清除浏览器缓存刷新无效
- **根因**：后端 `remote_nodes` 表已删除，但前端 `nodesStore` 持久化在 `localStorage` 的 `etamusic_nodes` key 中，浏览器「清除缓存」默认只清 HTTP 缓存不清 localStorage；且 `App.vue` 的 `refreshRemoteTokens` 缺少反向清理逻辑，不会把「前端有但后端没有」的节点移除
- **次因**：`cleanup_node_cache` 中 `ClientPlaylistItem.node_id` 用 `str(node_id)`（如 "5"）匹配，但写入时格式是 `f"remote-{node_id}"`（如 "remote-5"），字符串不匹配导致客户端播放列表引用残留，工作台持续展示离线节点曲目

### 设计要点

1. **App.vue refreshRemoteTokens 反向清理**
   - 拉取后端节点列表后，先用 `backendNodeIds = new Set(remoteNodes.map(row => \`remote-${row.id}\`))` 构建后端存在的节点 id 集合
   - 遍历 `nodesStore.nodes`，筛出 `id` 形如 `remote-{数字}` 但不在 `backendNodeIds` 中的节点，调用 `removeNode` 移除
   - 本地节点 `id` 是数字（`addNode` 不传 id 时自动生成），不会匹配 `remote-` 前缀，安全

2. **cleanup_node_cache 修复 node_id 格式 bug**
   - 改为 `ClientPlaylistItem.node_id.in_([f"remote-{node_id}", str(node_id)])` 同时清理新格式与旧残留
   - `synchronize_session=False` 避免批量 DELETE 后 session 与 DB 不一致

3. **delete_remote_node 执行顺序修正**
   - 原顺序：`db.delete(node) → db.commit() → cleanup_node_cache(db, node_id)`
   - 问题：若 cleanup 抛异常返回 500，DB 中节点已删，前端误以为删除失败但后端实际已无节点
   - 新顺序：`cleanup_node_cache(db, node_id) → db.delete(node) → db.commit()`
   - cleanup 幂等（重试时无匹配记录返回 0），失败时节点仍在，用户可重试

### 生产环境历史残留数据清理

修复部署后，对于历史已删除节点但 `client_playlist_items` 表中残留的引用，执行以下 SQL 清理（仅清理纯数字格式的旧残留 node_id）：

```sql
-- 先查待清理条目（确认无误后再执行 DELETE）
SELECT id, playlist_id, node_id, title FROM client_playlist_items
WHERE node_id GLOB '[0-9]*' AND node_id NOT LIKE 'remote-%' AND node_id NOT LIKE 'local%';
```

## 2026-07-21 — 插件前端动态加载架构改造（1.4.0）

### 需求

- **解决「未安装插件也打包进 dist」的架构问题**：1.3.1 之前所有插件前端代码静态 import 到主应用，无论用户是否启用都会下载到浏览器
- **支持插件独立分发**：每个插件自带前端 dist，主应用运行时根据启用的插件动态加载对应前端 bundle
- **新增插件无需重新部署主应用**：插件升级或新增只更新 `plugins/<name>/frontend/dist/`，主应用 dist 不变

### 设计要点

1. **运行时依赖暴露（`web/frontend/src/lib/runtime.js`）**
   - 主应用 `app.mount()` 前调用 `setupRuntime()`，将 vue / pinia / vueRouter / UI 组件库 / 核心 stores / axios / howler / utils / icons 暴露到 `window.__etamusic`

   - Vite library mode 输出 ESM 模块，`base: '/plugins-assets/<name>/'` 确保动态 import 的 chunk URL 正确

6. **主应用全局设置页解耦**
   - 原 `SettingsView.vue` 通过 `await import('../plugins/asmr_one/api')` 动态加载插件 API
   - 迁移后改为在 `api/plugin.js` 添加 `getAsmrSettings` / `updateAsmrSettings` / `getBiliSettings` / `updateBiliSettings`，主应用直接调用插件后端端点

## 2026-07-16 — 修复离线节点被误显示为"版本不兼容"

### 修复

#### 问题现象
- 生产环境 `music.yuzusoft.me` 上节点 `rognode.yuzusoft.me` 未启动时，节点管理页面状态显示为「● 版本不兼容」，实际应当显示「● 离线」。

#### 根因
- `web/frontend/src/stores/nodes.js` 的 `checkNodeVersion` 在 `GET /api/version` 网络失败时（节点未启动、连接超时等），把 catch 分支统一返回 `incompatible`，没有区分「网络不可达」和「真正版本不兼容」。
- `web/frontend/src/views/NodesView.vue` 的 `checkOneHealth` Step 1 因此直接 return，永远进不到 Step 2 登录验证分支，离线节点被错误标记为 `incompatible`。

#### 修复
- `nodes.js checkNodeVersion`：网络错误时不再返回 incompatible，而是清空旧校验结果并重新抛出异常，交由调用方处理。只有真正拿到 node 返回的版本信息且不匹配时才返回 incompatible。
- `NodesView.vue checkOneHealth`：Step 1 catch 到网络异常后继续执行 Step 2 登录验证，由登录结果决定最终状态（401/403 → auth_failed；其他网络错误 → offline）。
- `NodesView.vue saveRemoteNode`：版本校验网络错误时同样继续尝试登录，与 `checkOneHealth` 行为一致；错误提示改为具体原因（节点未启动/连接超时/接口不存在等，≤15字）。
- `App.vue refreshRemoteTokens`：应用启动时自动刷新远程节点 token 的流程存在同类缺陷——`checkNodeVersion` 网络错误被外层 catch 直接吞掉，不会继续 `loginRemoteNode`，导致离线节点的登录验证被跳过。改为内层 try-catch 单独捕获版本校验异常并继续登录流程，与 `checkOneHealth` 行为一致。
- 新增 `formatOfflineReason(e)` 辅助函数：根据 axios 错误码（ECONNABORTED / ERR_NETWORK / 404 / 502/503）归纳具体离线原因，便于用户反馈和排查。

#### 验证
- 本地启动访问端（后端 8000），配置一个指向未启动节点（127.0.0.1:8001）的远程节点。
- 用 Chrome headless + CDP 读取节点管理页表格 DOM，确认状态显示为「● 离线」而非「● 版本不兼容」。

## 2026-07-14 — 新建播放列表弹窗 + m3u 文件批量导入

### 概述

本次更新重构了新建播放列表的交互流程，新增 m3u/m3u8 播放列表文件导入能力，并优化了侧边栏管理功能的显示策略。

### 新增功能

#### 1. 新建播放列表改为弹窗交互
- **变更前**：点击曲库树分组旁的「+」按钮会直接创建名为「新建播放列表」的默认列表，需要双击重命名。
- **变更后**：点击「+」按钮弹出新建播放列表对话框，用户先输入名称再创建；本机分组仅支持空白播放列表，节点分组额外支持「从本机 m3u 文件导入」选项。
- 涉及文件：
  - [web/frontend/src/components/NewPlaylistDialog.vue](web/frontend/src/components/NewPlaylistDialog.vue)（新增）
  - [web/frontend/src/components/PlaylistTree.vue](web/frontend/src/components/PlaylistTree.vue)（+ 号按钮改为打开弹窗）

#### 2. 节点播放列表支持从 m3u 文件批量导入
- **功能说明**：在新建播放列表弹窗中选择「从本机 m3u 文件导入」模式后，可浏览节点所在主机的文件系统，选择 m3u/m3u8 文件或文件夹进行导入。
- **导入模式**：
  - **单个 m3u 文件**：将该 m3u 引用的所有音频文件按 m3u 顺序导入到指定监控目录，并创建一个播放列表（名为 m3u 文件名或自定义名称）。
  - **文件夹递归**：递归查找所选文件夹下所有 m3u/m3u8 文件，每个 m3u 文件创建一个独立播放列表（名为 m3u 文件名）。
- **复制/剪切**：可选择「复制」（保留原文件）或「剪切」（移动原文件）将音频文件入库到目标监控目录。
- **执行流程**：通过节点任务队列异步执行，依次完成：① 解析 m3u 引用并复制/剪切音频文件 → ② 扫描入库 → ③ 按原顺序创建播放列表与曲目项。任务进度可通过任务队列查看。
- **支持编码**：自动识别 UTF-8 BOM、UTF-8、GBK 编码的 m3u 文件。
- **冲突处理**：目标目录存在同名文件时自动追加 `_1`、`_2` 等后缀避免覆盖。
- 涉及文件：
  - [node/eta_node/m3u_importer.py](node/eta_node/m3u_importer.py)（新增，导入任务处理器）
  - [node/eta_node/task_handlers.py](node/eta_node/task_handlers.py)（注册 `import_m3u` 处理器）
  - [node/eta_node/routers/tasks.py](node/eta_node/routers/tasks.py)（`_ALLOWED_TASK_TYPES` 增加 `import_m3u`）
  - [node/eta_node/routers/watch_dirs.py](node/eta_node/routers/watch_dirs.py)（`browse_directory` 增加 `include_files` 参数，附带返回 m3u/m3u8 文件）
  - [web/frontend/src/api/node.js](web/frontend/src/api/node.js)（新增 `submitM3uImport`，`browseDirectories` 增加 `includeFiles` 参数）
