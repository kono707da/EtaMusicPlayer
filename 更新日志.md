# 更新日志

## 2026-07-20 — 网易云音乐插件后端集成（1.3.0）

### 需求

- **网易云音乐插件**：作为 eta_web 外部插件，提供扫码登录、搜索、播放、推荐、歌单克隆、多账号管理等功能
- **参考 YesPlayMusic 项目实现**：将 NeteaseCloudMusicApi 的核心加密逻辑翻译为 Python，不部署外部服务
- **eapi 加密替代 weapi**：weapi 二维码登录被风控拒绝（code=8821），改用 eapi AES-128-ECB 加密
- **多账号免扫码切换**：JSON 文件存储多个账号的 cookies，切换账号无需重新扫码
- **下载/克隆与节点查重解耦**：用户明确取消下载后自动查重，查重作为独立节点功能

### 设计要点

1. **插件架构（外部插件）**
   - 插件目录：`plugins/netease_music/eta_netease/`
   - 通过 `plugins/registry.py` 的 `PLUGIN_PACKAGE_MAP` 映射注册
   - `plugin.py` 的 `register(app)` 在 eta_web 启动时挂载路由
   - 账号数据目录：`web/backend/data/netease_accounts/`（首次启动自动创建）
   - 符合「部署后应用不含任何内置插件」原则，插件可通过在线安装或手动导入

2. **eapi 加密实现（`crypto.py`）**
   - AES-128-ECB 加密，密钥 `e82ckenh8dichen8`
   - 数据格式：`{url_path}-36cd479b6b5-{json}-36cd479b6b5-{md5_digest}`
   - 输出 hex 大写
   - 同时保留 weapi 加密实现（备用，目前未使用）

3. **HTTP 客户端（`client.py`）**
   - 全局共享 `requests.Session`，禁用系统代理
   - cookies 优先级：显式传入 > session 中已保存（登录后的 MUSIC_U）> 当前账号
   - eapi 模式下 cookie 在 set-cookie 响应头里，由 requests.Session 自动保存
   - PC 客户端伪装：UA `NeteaseMusicDesktop/3.0.18.203152`，os=pc，appver=3.1.17.204416
   - 提供 `NcmError` 异常类和 `check_response()` 工具函数

4. **多账号管理（`accounts.py`）**
   - JSON 文件存储：`accounts/<ncm_uid>.json`
   - 字段：`ncm_uid` / `nickname` / `avatar_url` / `vip_type` / `cookies` / `last_login_at` / `is_current`
   - 函数：`list_accounts` / `get_account` / `save_account` / `delete_account` / `switch_account` / `get_current_account` / `get_current_cookies`
   - `save_account` 自动设为当前账号；`switch_account` 加载对应 cookie 实现免扫码切换

5. **API 模块拆分**
   - `api_login.py`：二维码登录（生成 key / 状态轮询 / 登录态查询 / 刷新 / 登录成功处理）
   - `api_search.py`：搜索（cloudsearch / 热搜 / 搜索建议）+ 歌曲（详情 / 播放 URL / 下载 URL / 歌词）
   - `api_recommend.py`：推荐（个性化歌单 / 新歌 / 每日推荐 / 历史推荐）+ 排行榜
   - `api_playlist.py`：歌单管理（详情 / 全部曲目 / 收藏 / 创建 / 删除 / 添加曲目 / 删除曲目 / 重命名）+ 关注用户（关注列表 / 粉丝列表 / 用户详情 / 用户歌单）

6. **FastAPI 路由（`routers.py`）**
   - 路由前缀：`/api/netease`
   - 端点数量：40+ 个，覆盖多账号管理、登录、搜索、歌曲、推荐、排行榜、歌单管理、关注用户
   - 二维码登录支持 SSE 推送（`GET /login/qrcode/wait`，`text/event-stream`）
   - 二维码图片以 SVG 内联返回（`GET /login/qrcode/image`），无需前端额外依赖

7. **登录状态码处理**
   - 800=cookie 过期，801=等待扫码，802=已扫码待确认，803=登录成功，8821=风控拒绝
   - 登录成功后调 `login_status` 获取账号信息，`save_account` 持久化

## 2026-07-20 — 播放完成判定配置 + 命名冲突修复（1.2.3）

### 需求

- **节点播放完成判定配置**：允许用户按曲目时长区分音乐和广播剧，分别设置播放完成百分比。达到百分比后仅记统计不切歌，用于解决广播剧（长音频）听到 70% 就算听完的统计需求
- **修复 plugin.py 命名冲突 bug**（测试时发现）：`from eta_node.routers import settings` 覆盖了 `from eta_node.config import settings`，导致全新 DB 首次启动时创建 admin 用户访问 `settings.default_admin_password` 报 `AttributeError`。已部署环境因 admin 已存在未触发，但新环境部署会失败

### 设计要点

1. **配置存储：节点端各自配置**
   - 新增 `PlaybackSettings` 模型（`models.py`）：单行表 `playback_settings`（id=1），字段 `duration_threshold_seconds` / `music_complete_percent` / `broadcast_complete_percent`
   - `bootstrap()` 首次启动创建默认记录（900s / 90% / 70%）
   - 老节点升级到 1.2.3 后 `init_db` 的 `create_all` 自动建表，无需手动迁移

2. **后端 API（`routers/settings.py`）**
   - `GET /api/settings/playback`：任何登录用户可读（前端播放器需要读取配置计算阈值）
   - `PUT /api/settings/playback`：admin only，支持部分更新
   - 校验：`duration_threshold_seconds` 60~86400，`music_complete_percent` / `broadcast_complete_percent` 1~100
   - 老节点（<1.2.3）无此 API，前端拉取失败用默认值兜底

3. **前端播放器改造（`stores/player.js`）**
   - 新增 `_getPlaybackSettings(nodeId)`：带缓存 + 防并发，按节点缓存配置
   - 新增 `_updateCompleteThreshold()`：曲目 `onload` 后异步计算完成阈值（秒）= 时长 × 完成百分比
   - `completeReported` 标志位防重复上报
   - `tick()` 轮询检测进度达到阈值时上报 `complete`（仅一次，不切歌）
   - `onend()` 兜底：自然结束时若未上报过 complete 则上报
   - `next()`/`prev()`/`jumpTo()`：仅当未上报 complete 时才上报 `skip`（已 complete 后切歌不算 skip，避免广播剧听到 70% 后切走被记 skip）

4. **配置入口（`views/NodesView.vue`）**
   - 节点行新增「播放设置」按钮（`SlidersHorizontal` 图标，仅在线节点可点击）
   - 弹 modal 展示 3 个输入框：分界时长（分钟）、音乐完成%、广播剧完成%
   - 保存后调用 `playerStore.invalidatePlaybackSettings(nodeId)` 清缓存，下次播放重新拉取

5. **plugin.py 命名冲突修复**
   - `from eta_node.routers import settings as settings_router`（别名 import）
   - `sub_app.include_router(settings_router.router)`
   - 保留 `from eta_node.config import settings` 不被覆盖

## 2026-07-19 — 节点端口冲突检测 + 看板文案修正（1.2.2）

### 需求

- **eta_node 打包后的 exe 启动前检测端口冲突**：被 eta_node 自身占用自动杀掉旧进程重启；被其他程序占用报错退出，提示用户
- **数据看板「本周入库」文案误导**：实际是过去 7 天滚动窗口（`now - timedelta(days=7)`），不是自然周（周一 00:00 起）。周一凌晨看「本周入库 6」会误解为本周一已入库 6 首，实为上周一到上周日的入库量

### 设计要点

1. **节点端口冲突检测（eta_node）**
   - 新增 `eta_node/port_manager.py`：纯标准库实现，跨平台（Windows: netstat/tasklist/taskkill；POSIX: lsof/ss + kill）
   - `ensure_port_available(host, port)` 启动前检测：
     - 端口空闲 → 直接返回
     - 端口被 eta_node 自身占用（进程名 `eta_node.exe`/`eta_node`）→ taskkill 强杀，等待最多 10 秒端口释放
     - 端口被其他程序占用 → 打印占用进程 PID + 名称，`sys.exit(1)`
   - 端口空闲检测用 `connect_ex` 而非 `bind`（Windows 上 `SO_REUSEADDR` 会让 bind 检测失效）
   - `standalone.py` `__main__` 入口在 `uvicorn.run` 前调用 `ensure_port_available`

2. **看板文案修正（eta_web）**
   - `DashboardView.vue`：「本周入库」→「近 7 天入库」，文案与后端 `week_ago = now - timedelta(days=7)` 滚动窗口语义一致
   - 后端字段名 `tracks_imported_this_week` 不变（避免 API/前端字段重命名连锁改动）

## 2026-07-16 — 修复离线节点被误显示为"版本不兼容"

### 修复

#### 问题现象
- 生产环境 `music.yuzusoft.me` 上节点 `rognode.yuzusoft.me` 未启动时，节点管理页面状态显示为「● 版本不兼容」，实际应当显示「● 离线」。

#### 根因
- `web/frontend/src/stores/nodes.js` 的 `checkNodeVersion` 在 `GET /api/version` 网络失败时（节点未启动、连接超时等），把 catch 分支统一返回 `incompatible`，没有区分「网络不可达」和「真正版本不兼容」。
- `web/frontend/src/views/NodesView.vue` 的 `checkOneHealth` Step 1 因此直接 return，永远进不到 Step 2 登录验证分支，离线节点被错误标记为 `incompatible`。

#### 修复
- `nodes.js checkNodeVersion`：网络错误时不再返回 incompatible，而是清空旧校验结果并重新抛出异常，交由调用方处理。只有真正拿到 node 返回的版本信息且不匹配时才返回 incompatible。
- `NodesView.vue checkOneHealth`：Step 1 catch 到网络异常后继续执行 Step 2 登录验证，由登录结果决定最终状态（401/403 → auth_failed；其他网络错误 → offline）。
- `NodesView.vue saveRemoteNode`：版本校验网络错误时同样继续尝试登录，与 `checkOneHealth` 行为一致；错误提示改为具体原因（节点未启动/连接超时/接口不存在等，≤15字）。
- `App.vue refreshRemoteTokens`：应用启动时自动刷新远程节点 token 的流程存在同类缺陷——`checkNodeVersion` 网络错误被外层 catch 直接吞掉，不会继续 `loginRemoteNode`，导致离线节点的登录验证被跳过。改为内层 try-catch 单独捕获版本校验异常并继续登录流程，与 `checkOneHealth` 行为一致。
- 新增 `formatOfflineReason(e)` 辅助函数：根据 axios 错误码（ECONNABORTED / ERR_NETWORK / 404 / 502/503）归纳具体离线原因，便于用户反馈和排查。

#### 验证
- 本地启动访问端（后端 8000），配置一个指向未启动节点（127.0.0.1:8001）的远程节点。
- 用 Chrome headless + CDP 读取节点管理页表格 DOM，确认状态显示为「● 离线」而非「● 版本不兼容」。

## 2026-07-14 — 新建播放列表弹窗 + m3u 文件批量导入

### 概述

本次更新重构了新建播放列表的交互流程，新增 m3u/m3u8 播放列表文件导入能力，并优化了侧边栏管理功能的显示策略。

### 新增功能

#### 1. 新建播放列表改为弹窗交互
- **变更前**：点击曲库树分组旁的「+」按钮会直接创建名为「新建播放列表」的默认列表，需要双击重命名。
- **变更后**：点击「+」按钮弹出新建播放列表对话框，用户先输入名称再创建；本机分组仅支持空白播放列表，节点分组额外支持「从本机 m3u 文件导入」选项。
- 涉及文件：
  - [web/frontend/src/components/NewPlaylistDialog.vue](web/frontend/src/components/NewPlaylistDialog.vue)（新增）
  - [web/frontend/src/components/PlaylistTree.vue](web/frontend/src/components/PlaylistTree.vue)（+ 号按钮改为打开弹窗）

#### 2. 节点播放列表支持从 m3u 文件批量导入
- **功能说明**：在新建播放列表弹窗中选择「从本机 m3u 文件导入」模式后，可浏览节点所在主机的文件系统，选择 m3u/m3u8 文件或文件夹进行导入。
- **导入模式**：
  - **单个 m3u 文件**：将该 m3u 引用的所有音频文件按 m3u 顺序导入到指定监控目录，并创建一个播放列表（名为 m3u 文件名或自定义名称）。
  - **文件夹递归**：递归查找所选文件夹下所有 m3u/m3u8 文件，每个 m3u 文件创建一个独立播放列表（名为 m3u 文件名）。
- **复制/剪切**：可选择「复制」（保留原文件）或「剪切」（移动原文件）将音频文件入库到目标监控目录。
- **执行流程**：通过节点任务队列异步执行，依次完成：① 解析 m3u 引用并复制/剪切音频文件 → ② 扫描入库 → ③ 按原顺序创建播放列表与曲目项。任务进度可通过任务队列查看。
- **支持编码**：自动识别 UTF-8 BOM、UTF-8、GBK 编码的 m3u 文件。
- **冲突处理**：目标目录存在同名文件时自动追加 `_1`、`_2` 等后缀避免覆盖。
- 涉及文件：
  - [node/eta_node/m3u_importer.py](node/eta_node/m3u_importer.py)（新增，导入任务处理器）
  - [node/eta_node/task_handlers.py](node/eta_node/task_handlers.py)（注册 `import_m3u` 处理器）
  - [node/eta_node/routers/tasks.py](node/eta_node/routers/tasks.py)（`_ALLOWED_TASK_TYPES` 增加 `import_m3u`）
  - [node/eta_node/routers/watch_dirs.py](node/eta_node/routers/watch_dirs.py)（`browse_directory` 增加 `include_files` 参数，附带返回 m3u/m3u8 文件）
  - [web/frontend/src/api/node.js](web/frontend/src/api/node.js)（新增 `submitM3uImport`，`browseDirectories` 增加 `includeFiles` 参数）
